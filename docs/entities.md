## 1. User / UserProfile

Базовый объект пользователя. Возвращается в теле `AuthResponse`, а также из `GET /me` и `GET /admin/users`.

| Поле | Тип | Обязательное | Описание |
|------|-----|:---:|---------|
| `id` | `integer` | ✅ | Уникальный идентификатор пользователя в БД |
| `full_name` | `string` | ✅ | Полное имя (минимум 2 символа при регистрации) |
| `email` | `string (email)` | ✅ | Email — используется как логин. Уникален в системе |
| `role` | `string (enum)` | ✅ | Роль: `employee` — обычный сотрудник, `admin` — администратор. При регистрации всегда `employee` |
| `coin_balance` | `integer` | ✅ | Текущий баланс монет. Не может быть отрицательным |
| `office` | `string` | ❌ | Офис сотрудника (например: `Казань`, `Ульяновск`). Необязательное поле |

**Пример:**
```json
{
  "id": 5,
  "full_name": "Иван Иванов",
  "email": "ivan@simbirsoft.com",
  "role": "employee",
  "coin_balance": 1500,
  "office": "Казань"
}
```

## 2. AuthResponse

Возвращается при успешном логине (`POST /auth/login`), регистрации (`POST /auth/register`) и обновлении токена (`POST /auth/refresh`).

| Поле | Тип | Описание |
|------|-----|----------|
| `access_token` | `string (JWT)` | Короткоживущий токен для авторизации запросов. TTL — **15 минут**. Передаётся в заголовке `Authorization: Bearer <token>` |
| `refresh_token` | `string` | Долгоживущий токен для получения нового `access_token`. TTL — **30 дней**. После использования инвалидируется |
| `user` | `UserProfile` | Данные авторизованного пользователя (см. секцию 1) |

> **Хранение на фронте:** оба токена хранятся в `sessionStorage`. При получении `401` axios-interceptor автоматически вызывает `POST /auth/refresh` и повторяет упавший запрос.

## 3. CoinTransaction

Запись об изменении баланса монет пользователя. Возвращается из `GET /me/coins` и `GET /admin/users/{id}/transactions`.

| Поле | Тип | Nullable | Описание |
|------|-----|:---:|---------|
| `id` | `integer` | ❌ | Идентификатор транзакции |
| `type` | `string (enum)` | ❌ | Тип операции: `manual` — ручное начисление/списание администратором; `order` — списание при оформлении заказа; `refund` — возврат монет |
| `amount` | `integer` | ❌ | Сумма изменения. Положительное — начисление, отрицательное — списание |
| `balance` | `integer` | ❌ | Баланс пользователя **после** этой транзакции |
| `comment` | `string` | ❌ | Комментарий к операции (причина начисления или списания) |
| `admin_name` | `string` | ✅ | Имя администратора, создавшего транзакцию. `null` для автоматических операций (`order`, `refund`) |
| `order_id` | `integer` | ✅ | ID заказа, с которым связана транзакция. Заполнено только для типа `order` и `refund` |
| `created_at` | `string (datetime)` | ❌ | Дата и время создания транзакции в формате ISO 8601 |

**Пример:**
```json
{
  "id": 101,
  "type": "manual",
  "amount": 500,
  "balance": 2000,
  "comment": "Премия за квартал",
  "admin_name": "Алексей Тимлид",
  "order_id": null,
  "created_at": "2024-03-15T10:30:00Z"
}
```

## 4. ProfileStats

Сводная статистика пользователя. Возвращается из `GET /me/stats`.

| Поле | Тип | Описание |
|------|-----|---------|
| `orders_count` | `integer` | Общее количество оформленных заказов |
| `coins_spent` | `integer` | Суммарное количество монет, потраченных на заказы |
| `coins_earned` | `integer` | Суммарное количество монет, полученных (начисленных вручную и через refund) |

## 5. Category

Категория товаров в каталоге. Возвращается из `GET /categories` и операций admin.

| Поле | Тип | Nullable | Описание |
|------|-----|:---:|---------|
| `id` | `integer` | ❌ | Уникальный идентификатор категории |
| `name` | `string` | ❌ | Отображаемое название категории (например: `Одежда`) |
| `code` | `string` | ❌ | Уникальный строковый ключ в формате `[a-z0-9-]+` (например: `clothes`, `mugs`). Используется в URL и фильтрации |
| `icon` | `string` | ✅ | Эмодзи или символ для отображения рядом с названием (например: `👕`) |
| `description` | `string` | ✅ | Текстовое описание категории |

**Пример:**
```json
{
  "id": 1,
  "name": "Одежда",
  "code": "clothes",
  "icon": "👕",
  "description": "Футболки, толстовки и другая одежда с символикой"
}
```

## 6. Product

Товар в каталоге. Возвращается из `GET /products` и `GET /products/{id}`.

| Поле | Тип | Nullable | Описание |
|------|-----|:---:|---------|
| `id` | `integer` | ❌ | Уникальный идентификатор товара |
| `name` | `string` | ❌ | Название товара |
| `description` | `string` | ❌ | Описание товара |
| `category_id` | `integer` | ❌ | ID категории, к которой относится товар |
| `category_name` | `string` | ❌ | Название категории (денормализованное поле для удобства фронта) |
| `category_code` | `string` | ❌ | Код категории (денормализованное поле) |
| `price_coins` | `integer` | ❌ | Базовая цена товара в монетах. Для товаров с вариантами может быть переопределена на уровне варианта |
| `has_variants` | `boolean` | ❌ | `true` — товар имеет варианты (размер, цвет). При добавлении в корзину **обязательно** передавать `variant_id` |
| `image_url` | `string` | ✅ | URL изображения товара. `null` если изображение не задано |
| `is_active` | `boolean` | ❌ | `true` — товар активен и виден в каталоге. При `false` — скрыт (soft delete) |
| `variants` | `array<ProductVariant>` | ❌ | Массив вариантов товара. Пустой массив `[]` если `has_variants: false` |

> **Важно:** если `has_variants: true` и `variants` пустой — товар добавить в корзину нельзя, нет доступных вариантов.

## 7. ProductVariant

Вариант товара (например: размер M, цвет белый). Вложен в объект `Product`.

| Поле | Тип | Nullable | Описание |
|------|-----|:---:|---------|
| `id` | `integer` | ❌ | Уникальный идентификатор варианта. Передаётся как `variant_id` при добавлении в корзину |
| `size` | `string` | ✅ | Размер (например: `S`, `M`, `L`, `XL`). `null` если размер не применим |
| `color` | `string` | ✅ | Цвет (например: `белый`, `чёрный`). `null` если цвет не применим |
| `sku` | `string` | ✅ | Артикул варианта (например: `HOODIE-M-WHT`). Используется для складского учёта |
| `price_coins` | `integer` | ✅ | Цена варианта в монетах. Если `null` — используется базовая цена из `Product.price_coins` |
| `in_stock` | `boolean` | ❌ | `true` — вариант доступен для заказа. `false` — отображать как недоступный |

**Пример:**
```json
{
  "id": 2,
  "size": "M",
  "color": "белый",
  "sku": "HOODIE-M-WHT",
  "price_coins": 500,
  "in_stock": true
}
```

## 8. CartItem

Позиция в корзине пользователя.

| Поле | Тип | Nullable | Описание |
|------|-----|:---:|---------|
| `id` | `integer` | ❌ | Идентификатор позиции в корзине (не ID товара) |
| `product_id` | `integer` | ❌ | ID товара |
| `product_name` | `string` | ❌ | Название товара (денормализовано для отображения) |
| `variant_id` | `integer` | ✅ | ID варианта. `null` для товаров без вариантов |
| `variant_label` | `string` | ✅ | Человекочитаемое описание варианта (например: `M / белый`). `null` для товаров без вариантов |
| `price_coins` | `integer` | ❌ | Цена **одной единицы** этой позиции на момент добавления в корзину |
| `quantity` | `integer` | ❌ | Количество единиц. Минимум `1` |

## 9. Cart

Корзина текущего пользователя. Возвращается из `GET /cart` и всех операций изменения корзины.

| Поле | Тип | Описание |
|------|-----|---------|
| `items` | `array<CartItem>` | Список позиций в корзине. Пустой массив `[]` если корзина пуста |
| `total_coins` | `integer` | Итоговая стоимость всех позиций (`сумма price_coins × quantity`). `0` если корзина пуста |

## 10. Order

Заказ пользователя. Базовый объект — возвращается в списке `GET /orders`.

| Поле | Тип | Описание |
|------|-----|---------|
| `id` | `integer` | Уникальный идентификатор заказа |
| `status` | `string (enum)` | Статус заказа (см. таблицу жизненного цикла ниже) |
| `total_coins` | `integer` | Итоговая сумма заказа в монетах на момент оформления |
| `created_at` | `string (datetime)` | Дата и время оформления заказа (ISO 8601) |

### Жизненный цикл статусов заказа

| Статус | Значение |
|--------|---------|
| `new` | Заказ только что оформлен, ещё не взят в работу |
| `pending` | Заказ на рассмотрении |
| `confirmed` | Заказ подтверждён администратором |
| `ready` | Заказ готов к выдаче |
| `done` | Заказ выдан сотруднику |
| `completed` | Заказ завершён |
| `cancelled` | Заказ отменён |

> Статус меняет только администратор через `PATCH /admin/orders/{id}/status`.

### OrderDetail

Расширенная версия `Order` — возвращается из `GET /orders/{id}` и `POST /orders/checkout`. Добавляет поле:

| Поле | Тип | Описание |
|------|-----|---------|
| `items` | `array<CartItem>` | Состав заказа — снимок корзины на момент оформления |

| Поле | Тип | Nullable | Описание |
|------|-----|:---:|---------|
| `item_id` | `integer` | ❌ | Идентификатор позиции заказа |

## 12. AdminOrder

Расширенная версия `Order` для административного контекста. Возвращается из `GET /admin/orders`. Содержит все поля `Order`, плюс:

| Поле | Тип | Описание |
|------|-----|---------|
| `user_id` | `integer` | ID пользователя, оформившего заказ |
| `user_name` | `string` | Полное имя пользователя (денормализовано для отображения в списке) |

## 13. ErrorResponse

Единый формат ошибок для всех эндпоинтов.

| Поле | Тип | Nullable | Описание |
|------|-----|:---:|---------|
| `error` | `string` | ❌ | Машиночитаемый код ошибки |
| `details` | `string` | ✅ | Человекочитаемое описание. Может отсутствовать |

### Коды ошибок

| Код | HTTP | Когда возникает |
|-----|:----:|----------------|
| `unauthorized` | 401 | Токен не передан, невалидный или истёк |
| `forbidden` | 403 | Нет прав (требуется роль `admin`) |
| `not_found` | 404 | Запрошенный ресурс не существует |
| `email_taken` | 409 | Email уже зарегистрирован в системе |
| `insufficient_balance` | 422 | Недостаточно монет для оформления заказа |
| `bad_request` | 400 | Ошибка валидации тела запроса |
| `db_error` | 500 | Внутренняя ошибка базы данных |

**Пример:**
```json
{
  "error": "insufficient_balance",
  "details": "need 1500 coins, have 300"
}
```
