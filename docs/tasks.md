# SimbirStore — Техническое задание на разработку бэкенда

> **Проект:** SimbirStore — внутренний корпоративный магазин товаров за монеты  
> **Стек:** Go, PostgreSQL, Vue 3 (фронт готов)  
> **Репозиторий:** https://gitlab.simbirsoft.com/backend/training/simbirstore/skeleton  
> **Документация по API:** `docs/api-reference.md`, `docs/entities.md`, `docs/swagger.yaml`

## Задачи первого приоритета

Задачи 0–3 нужно сделать в первую очередь — они являются зависимостями для большинства остальных задач. Задачу 4 (каталог) можно брать параллельно, она не требует авторизации.

| # | Задача | Почему в приоритете |
|---|--------|---------------------|

## Задача 0 — Подключение к базе данных

**Issue:** #15  
**Блокирует:** все задачи

Бэкенд должен читать `DATABASE_URL` из переменных окружения и устанавливать соединение с PostgreSQL при старте. Если соединение не удалось — сервер не запускается.

**Критерии приёмки:**
- `go run .` запускается без ошибок, в консоли видно подтверждение подключения к БД
- `GET /api/ping` возвращает статус соединения с БД в ответе
- Если `DATABASE_URL` не задан — сервер падает с понятной ошибкой

## Задача 1 — Регистрация пользователя

**Эндпоинт:** `POST /api/register`  
**Блокирует:** задачи 2, 3 и всё что связано с пользователями

Новый пользователь регистрируется по email и паролю. Пароль хранится в БД как bcrypt-хэш. При успехе возвращается пара токенов и данные пользователя.

**Что нужно:**
- Принять тело запроса: `email`, `password`, `full_name`, `office` (необязательно)
- Провалидировать поля (формат email, минимальная длина пароля и имени)
- Сохранить пользователя в таблицу `users` с ролью `employee`
- Вернуть `access_token`, `refresh_token`, объект `user`

**Критерии приёмки:**
- Успешная регистрация → `201` с токенами
- Повторный email → `409` с `error: "email_taken"`
- Невалидные поля → `400`
- Пароль в БД хранится как bcrypt-хэш, не plain text

## Задача 2 — Авторизация (логин и логаут)

**Эндпоинты:** `POST /api/login`, `POST /api/logout`, `POST /api/refresh`  
**Зависит от:** задачи 0, 1  
**Блокирует:** задачу 3 и все защищённые эндпоинты

Пользователь входит по email + паролю, получает пару токенов. Refresh-токен хранится в БД для ротации.

**Что нужно:**
- `/login` — проверить пароль, вернуть `access_token` + `refresh_token`
- `/logout` — инвалидировать refresh-токен в БД
- `/refresh` — выдать новую пару токенов, старый refresh становится недействительным (ротация)

**Критерии приёмки:**
- Неверный пароль → `401`
- После `/logout` старый refresh-токен не работает
- После `/refresh` старый refresh-токен не работает, выдан новый
- `access_token` живёт 15 минут, `refresh_token` — 30 дней

## Задача 3 — JWT Middleware

**Зависит от:** задачи 2  
**Блокирует:** все защищённые эндпоинты (задачи 5–10)

Middleware проверяет заголовок `Authorization: Bearer <token>`, валидирует подпись JWT и кладёт данные пользователя в контекст запроса.

**Что нужно:**
- Извлечь токен из заголовка `Authorization`
- Проверить подпись с помощью `JWT_SECRET` из окружения
- При успехе — положить `user_id` и `role` в `context`
- Реализовать отдельную проверку роли `admin` для защиты `/api/admin/*`

**Критерии приёмки:**
- Запрос без токена → `401`
- Просроченный токен → `401`
- Токен с ролью `employee` на `/api/admin/*` → `403`
- Данные пользователя доступны из контекста в следующих хэндлерах

## Задача 4 — Каталог товаров

**Эндпоинты:** `GET /api/categories`, `GET /api/products`, `GET /api/products/{id}`  
**Зависит от:** задачи 0  
**Не требует авторизации**

Публичное API для просмотра каталога. Товары можно фильтровать по категории и искать по названию.

**Что нужно:**
- Список категорий из таблицы `categories`
- Список товаров с фильтрацией по `category` (code) и поиском по `search` (без учёта регистра)
- Пагинация через `limit` и `offset`
- Неактивные товары (`is_active = false`) не попадают в выдачу
- Каждый товар включает список своих вариантов

**Критерии приёмки:**
- `GET /api/products?category=clothes&search=футболка` работает корректно
- Товары с `is_active = false` не видны
- При пустой выборке возвращается `{"items": []}`, а не `null`

## Задача 5 — Профиль пользователя

**Эндпоинты:** `GET /api/me`, `GET /api/me/coins`, `GET /api/me/stats`  
**Зависит от:** задач 2, 3

Авторизованный пользователь видит свой профиль, баланс монет и историю транзакций.

**Что нужно:**
- `/me` — вернуть данные текущего пользователя из таблицы `users`
- `/me/coins` — история транзакций с монетами (тип, сумма, баланс после, дата)
- `/me/stats` — агрегаты: количество заказов, потрачено монет, заработано монет

**Критерии приёмки:**
- Все три эндпоинта требуют валидный JWT
- `coin_balance` совпадает с суммой транзакций в истории
- При отсутствии данных поля возвращают `0`, не `null`

## Задача 6 — Корзина

**Эндпоинты:** `GET /api/cart`, `POST /api/cart/items`, `PATCH /api/cart/items/{id}`, `DELETE /api/cart/items/{id}`  
**Зависит от:** задач 2, 3, 4

Каждый авторизованный пользователь имеет одну корзину. Корзина создаётся автоматически при первом обращении.

**Что нужно:**
- Добавление товара — если товар уже есть, увеличить `quantity`
- Для товаров с вариантами `variant_id` обязателен
- Изменение количества через PATCH
- Удаление позиции
- Ответ всегда содержит актуальное состояние корзины с `total_coins`

**Критерии приёмки:**
- Повторное добавление того же товара увеличивает quantity, а не создаёт дубликат
- Товар с `has_variants: true` без `variant_id` → `400`
- Позиции чужой корзины недоступны → `404`

## Задача 7 — Заказы

**Эндпоинты:** `POST /api/orders/checkout`, `GET /api/orders`, `GET /api/orders/{id}`  
**Зависит от:** задач 2, 3, 5, 6

Пользователь оформляет заказ из корзины. Монеты списываются атомарно вместе с созданием заказа.

**Что нужно:**
- Проверить что корзина не пуста
- Проверить что баланса хватает
- В одной транзакции: создать заказ, списать монеты, записать транзакцию, очистить корзину
- История заказов текущего пользователя
- Детали конкретного заказа

**Критерии приёмки:**
- Пустая корзина → `400`
- Недостаточно монет → `402`
- После успешного checkout: корзина пуста, баланс уменьшен, заказ создан со статусом `new`
- Если что-то упало посередине — ни корзина, ни баланс, ни заказ не изменились (транзакция БД)

## Задача 8 — Админ: управление заказами

**Эндпоинты:** `GET /api/admin/orders`, `GET /api/admin/orders/{id}/items`, `PATCH /api/admin/orders/{id}/status`  
**Зависит от:** задач 3, 7

Администратор видит все заказы всех пользователей и управляет их статусом.

**Что нужно:**
- Список всех заказов с возможностью фильтрации по `status`
- Состав конкретного заказа
- Изменение статуса заказа (допустимые значения: `new`, `pending`, `confirmed`, `ready`, `done`, `completed`, `cancelled`)

**Критерии приёмки:**
- Доступно только роли `admin`, иначе `403`
- Недопустимый статус → `400`
- Несуществующий заказ → `404`

## Задача 9 — Админ: управление товарами и категориями

**Эндпоинты:** `POST/PUT/DELETE /api/admin/products`, `POST/PUT/DELETE /api/admin/products/{id}/variants`, `POST/PUT/DELETE /api/admin/categories`  
**Зависит от:** задач 3, 4

Администратор создаёт, редактирует и удаляет товары, варианты и категории.

**Что нужно:**
- CRUD для товаров (удаление — soft delete через `is_active = false`)
- CRUD для вариантов товара
- CRUD для категорий
- При добавлении первого варианта — автоматически выставить `has_variants = true`
- При удалении последнего варианта — сбросить `has_variants = false`
- Удаление категории с привязанными товарами → `409`

**Критерии приёмки:**
- Удалённый товар не виден в каталоге, но не удалён из БД
- Дублирующий `code` категории → `409`

---

## Задача 10 — Админ: управление пользователями и монетами

**Эндпоинты:** `GET /api/admin/users`, `POST /api/admin/users/coins`, `GET /api/admin/users/{id}/transactions`  
**Зависит от:** задач 3, 5

Администратор видит всех пользователей и может начислять или списывать монеты.

**Что нужно:**
- Список всех пользователей
- Начисление/списание монет с комментарием
- История транзакций конкретного пользователя

**Критерии приёмки:**
- Списание не уводит баланс в минус → `400`
- Несуществующий пользователь → `404`
- Транзакция записывается в `coin_transactions`
